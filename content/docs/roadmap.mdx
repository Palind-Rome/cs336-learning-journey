## 先看全景

一段原始文本先被 **tokenizer** 变成整数；Transformer 根据前文预测下一个整数；训练循环用交叉熵和 AdamW 更新参数。模型变大后，瓶颈转向 GPU kernel、显存和通信；要决定“多大模型、多少数据”，需要缩放律；数据要经过抽取、过滤、去重与混合；预训练完成后，再用 SFT 和偏好/奖励信号教模型更好地回答。

```text
网页/WET → 清洗与去重 → BPE token → Transformer → 预训练
                                               ↓
用户问题 ← 安全与偏好 ← SFT / DPO / GRPO ← 评测
```

## 每个模块的“毕业标准”

| 模块 | 你应当能回答 | 作品 |
|---|---|---|
| 基础 | BPE 为什么有效？attention 的四个轴是什么？ | 能训练一个 TinyStories 小模型 |
| 系统 | FLOPs、HBM、通信哪个是瓶颈？ | 能解释 profile 并实现正确 kernel |
| 缩放 | 固定预算下 N 与 D 怎么选？ | 能拟合并外推一条 power law |
| 数据 | 为什么重复文本会改变模型行为？ | 可审计的过滤、去重流水线 |
| 对齐 | SFT、DPO、GRPO 分别用了什么监督？ | 可复现的评测与训练对比 |

## 学习纪律

每次实验保存 `config + git commit + 指标 + 一句话结论`。若结果异常，排查顺序是：随机种子和数据 → 张量形状与 mask → 数值稳定性 → 优化器/学习率 → 性能。不要一开始就把问题归咎于“GPU 玄学”。
