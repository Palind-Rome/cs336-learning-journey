## 0. 三个量先不要混

- $N$：通常指 non-embedding parameters；
- $D$：训练处理的 token 总数；
- $C$：训练 FLOPs，Transformer 粗略近似 $C=6ND$。

系数 6 的直觉是：每个 parameter 对每个 token，forward 大约贡献 2 次浮点操作，backward 计算 activation gradient 与 weight gradient 又约是 forward 的 2 倍，总计约 $2+4=6$。这是跨架构比较的粗略账本，不是墙钟时间预测器；不同 width/depth、batch、dtype、kernel 与硬件利用率即使 $6ND$ 相同，runtime 也可能不同。

固定 C 时，模型过小会欠拟合，数据再多也受 capacity 限制；模型过大时 $D=C/(6N)$ 太少，gradient steps 不足，模型没有训练充分。因此 loss 对 log N 常呈 U 形，最低点就是该 compute budget 的 `Nopt`。

## 1. 用合成数据完整复现 IsoFLOPs

<SourceBrief label="题面译文" title="Scaling laws 与 IsoFLOPs：完整题目">

本节复习 Chinchilla 论文的一种拟合方法。核心问题是：给定训练大语言模型的 compute budget C，选择什么 model size、training tokens 等 hyperparameters 能得到最低 training loss？主要挑战是怎样从较小规模实验外推到更大规模。第二部分可以加入 Kaplan 等人与 Tensor Programs V 等其他资料的方法。

#### 2.1 IsoFLOPs profiles

有 N 个参数、D 个数据 tokens 的 Transformer，训练 compute 近似 $C=6ND$。Chinchilla 的 IsoFLOPs 方法为：对每个 C，训练不同 N 的模型，同时令 $D=C/(6N)$，得到各自 final training loss L。

这样会产生多组 runs：同一组使用相同 $C_i$，但 model size $N_{ij}$ 不同。Hoffmann 等人经验上观察到，在固定 $C_i$ 时，final loss 与 model size 近似二次关系。直觉是：N 极小时，无论投入多少 compute，模型都没有 capacity 拟合数据，loss 很高；N 增大时 loss 平滑下降；超过某点后，模型太大，固定 C 无法提供足够 gradient steps，训练不充分，loss 又升高。极端地，$N\to\infty$ 时连一步 gradient update 都超过 C，训练会在很高 loss 停止。

拟合流程先在每个 $C_i$ 找 optimal model/data size，再拟合 power law，用比已有 runs 更大的 target C 预测 N/D。对每个 IsoFLOPs profile，选择 loss 最低的 $N_{opt}(C_i)$，得到成对数据 $\langle C_i,N_{opt}(C_i)\rangle$；Dopt 同理。拟合 $N_{opt}\propto C^a$、$D_{opt}\propto C^b$，再外推目标预算。

`chinchilla_isoflops` 题要求用 `data/isoflops_curves.json` 的 synthetic runs 复现。JSON 是对象数组，每个对象含 `parameters`、`compute_budget`、`final_loss`。可用 scipy，尤其 `scipy.optimize.curve_fit`，也可用其他拟合方法。虽然原论文对每条 profile 拟合 quadratic 再取 minimum，本题建议直接选择最低 training loss 的观测 run。

(a) 展示拟合使用的 $\langle C_i,N_{opt}(C_i)\rangle$ 与外推 compute-optimal N。回答 $10^{23}$ 与 $10^{24}$ FLOPs 的预测 model size。提交一张 model size 对 compute budget 的 scaling-law 图，既有 fit points，也至少外推到 $10^{24}$，并用一句话写预测。

(b) 用 $\langle C_i,D_{opt}(C_i)\rangle$ 做 dataset size scaling law，回答 $10^{23}$ 与 $10^{24}$ FLOPs 的 predicted dataset size。提交相应图与一句话预测。

</SourceBrief>

官方 `data/isoflops_curves.json` 有 72 个 run、9 档 compute，每档 8 个不同 N。对象字段是 `parameters`、`compute_budget`、`final_loss`。

### 1.1 第一步：按 C 分组并直接取最低 loss

题面说明 Chinchilla 原论文会对每条 profile 拟合二次曲线；本题推荐直接选择观测到的最低点。这样不需要假设每档 8 个点恰好落在对称抛物线上：

```python
def select_isoflops_minima(runs):
    grouped = {}
    for run in runs:
        grouped.setdefault(float(run['compute_budget']), []).append(run)
    return [
        min(grouped[compute], key=lambda run: run['final_loss'])
        for compute in sorted(grouped)
    ]
```

### 1.2 第二步：在 log 空间拟合幂律

设：

$$N_{opt}(C)=A C^a,\qquad D_{opt}(C)=B C^b.$$

取对数：

$$\log N_{opt}=\log A+a\log C.$$

使用 ordinary least squares：

```python
def fit_power_law(x, y):
    design = np.column_stack([np.ones_like(x), np.log(x)])
    log_coefficient, exponent = np.linalg.lstsq(
        design, np.log(y), rcond=None
    )[0]
    return np.exp(log_coefficient), exponent
```

完整脚本见 [`fit_isoflops.py`](https://github.com/Palind-Rome/cs336-learning-journey/blob/main/examples/assignment3/fit_isoflops.py)，会自动选 minima、拟合、生成两张 log–log 图并打印预测：

```bash

```

## 2. 读懂真实训练 API

<SourceBrief label="题面译文" title="Training API、实验设计与 leaderboard：完整题目">

本节在固定探索预算下，通过 training API 查询真实 runs 来拟合。主要目标是选择在 48 B200-hours 上 validation loss 最低的 model size 与 hyperparameters，并预测其 loss。为了设置目标规模的 hyperparameters，建议先分析小规模中各 hyperparameter 如何影响 validation loss。开始前必须仔细计划；超过 12 B200-hours 后 API 会拒绝更多请求。

#### 3.1 Training run 细节

API 在 B200 GPU 上运行真实训练。提交后 experiment 进入共享 queue。queued 与 running experiments 会按各自完整 `max_runtime_seconds` 预留 12 小时预算。完成或失败后，API 用报告的实际 runtime 重算费用，并把它限制在至少 1 秒、至多 max runtime。若一个预留 30 分钟的 run 在 12 分钟完成，18 分钟归还；timeout 收取完整 max runtime。

每个 job 训练 `n_evals` 个 chunks，每块处理 `total_train_tokens/n_evals` tokens，随后验证，并把 loss 序列放在 `status.val_losses`。completed experiment 的 final validation loss 是列表最后一项。timeout failed experiment 可能返回 partial losses，但它不是 completed run。

数据顺序固定。training data 是最多 500B tokens 的 tokenized DCLM，不做 epoching；每个配置消费同一确定性 token 顺序。validation set 使用 $2^{18}$ 个 validation tokens，final leaderboard 会用更多。`model_seed` 只控制 model initialization，不控制 data order。

模型架构基本和 A1 相同，有少量差异；仓库提供源码参考。training code 仅供阅读，只能通过 API 训练。数据词表 32K，context 512。API 支持 AdamW 与 SGD，均有 warmup-cosine schedule。示例用 AdamW、weight decay 0.01、gradient clipping 1.0、warmup fraction 0.05、final LR fraction 0.1。训练 loss 是 512-token sequences 上的 mean next-token cross entropy；模型无 dropout。

#### 3.2 Training API

API key 是 8 位 Stanford ID，例如 06123456。必须位于 Stanford network，可能需要 VPN。可以访问 `http://hyperturing.stanford.edu:8000/docs` 做连通性检查，并用 `/budget` 验证 key。所有 request 的 header 都应含 `X-API-Key`。

`POST /submit` 接受一个 training config 并把 run 加入 queue，返回 experiment id。request body 字段如下：

- `architecture_config`：含 `attention_bias`、`head_dim`、`hidden_size`、`intermediate_size`、`num_attention_heads`、`num_hidden_layers`、`num_key_value_heads`、`rms_norm_eps`、`rope_theta`、`tie_word_embeddings`、`dtype`、`vocab_size`；dtype 只能是 `float32` 或 `bfloat16`；
- `optimizer_config`：AdamW 或 SGD 配置，题面示例使用 AdamW；
- `train_batch_size`、`val_batch_size`：整数 batch sizes；
- `n_evals`：训练期间 validation 次数；
- `total_train_tokens`：总训练 token；
- `max_runtime_seconds`：预留的最大墙钟时间，1 秒到 12 小时；
- `model_seed`：初始化随机种子。

API 固定 `seq_len=512`，validation tokens 为 $2^{18}\approx262K$，并检查一致性：hidden size 等于 heads×head dim；attention heads 能被 KV heads 整除；total tokens 能被 `512*train_batch_size` 整除等。

response 含 `experiment_id` 与 `budget_summary`；后者含 used、remaining、total seconds。同一 config 重复提交返回 409，不额外预留预算；max runtime 大于 remaining budget 返回 400。

题面完整示例 architecture 为 head_dim 64、hidden 448、intermediate 1280、7 heads、9 layers、7 KV heads、RMS epsilon 1e-6、RoPE theta 1,000,000、untied embeddings、BF16、vocab 32,000。optimizer scheduler peak LR 3e-4、final fraction 0.1、warmup 0.05、initial 0；weight decay 1e-2、betas 0.9/0.95、eps 与 eps_root 1e-8、grad clip 1.0。train batch 128、val batch 32、16 evals、1,048,576 tokens、max runtime 30 秒、seed 0。POST 成功示例返回 experiment 1，used 30、remaining 43170、total 43200 秒。

`GET /budget` 返回当前 key 的 `used_seconds`（已用或已预留）、`remaining_seconds`、`total_budget_seconds`。

`GET /experiments` 返回当前 key 提交的全部 experiments。每个对象含 id、training config 与 status。`queued` 表示等待；`running` 表示运行，并可能含目前观测的 val losses。running 状态包含 startup 与 preemption，所以显示时间可能超过 max runtime。`completed` 含 actual used runtime、val losses 与 completion time。`failed` 表示 crash 或 timeout；timeout 可能含 partial losses。

`GET /experiment/{experiment_id}` 返回单个 experiment，格式与列表 entry 相同。应轮询直到 `status.status_type == "completed"`，并把 `status.val_losses[-1]` 作为该 completed run 的 final validation loss。

`POST /final_submission` 提交 predicted optimal `training_config` 与 `predicted_final_loss`。可重复提交，最新一次覆盖旧 submission。`GET /final_submission` 返回当前 submission；若从未提交则为 None。

#### 3.3 Leaderboard / `scaling_laws`

现在用 API 为 48 B200-hour run 选择 model config，目标是让最终 validation loss 最低。

### 2.1 budget 是“预留”，完成后才按实际结算

`POST /submit` 后，queued/running experiment 立即用完整 `max_runtime_seconds` 占用 12 小时预算。完成或失败时，API 才把计费改为实际 runtime，并裁剪在 1 秒与 max runtime 之间；若 timeout，则收满 max runtime。因此不能一次提交很多夸大的 timeout，把全部预算锁住。

提交前先读 `/budget`：

```python
def submit(self, config):
    reserve = float(config['max_runtime_seconds'])
    remaining = float(self.budget()['remaining_seconds'])
    if reserve > remaining:
        raise RuntimeError(f'need {reserve}s, only {remaining}s remains')
    return self._request('POST', '/submit', json=config)
```

相同 config 重复提交返回 HTTP 409，且不会再次占预算。要复现同一超参数，应更换 `model_seed`，并把 seed 明确纳入 config。

### 2.2 一个合法 config 要满足什么

architecture 必须包含 attention bias、head dim、hidden size、intermediate size、attention heads、hidden layers、KV heads、RMS epsilon、RoPE theta、是否 tied embeddings、dtype 与 vocab。重要约束：

- `hidden_size == num_attention_heads * head_dim`；
- attention heads 能被 KV heads 整除；
- dtype 只能是 `float32` 或 `bfloat16`；
- context length 固定 512、vocab 固定 32,000；
- `total_train_tokens` 必须被 `512 * train_batch_size` 整除；
- max runtime 为 1 秒到 12 小时；
- mean next-token CE 在 512-token sequences 上计算；模型没有 dropout。

配套 [`architecture.py`](https://github.com/Palind-Rome/cs336-learning-journey/blob/main/examples/assignment3/architecture.py) 把连续 Nopt 映射到合法离散 shape。题面估算 non-embedding parameters 为：

$$N\approx12\;n_{layers}\;d_{model}^2.$$

```python
def nearest_architecture(target_parameters):
    candidates = []
    for hidden in range(256, 8193, 64):
        for layers in range(4, 129):
            estimate = 12 * layers * hidden**2
            error = abs(math.log(estimate / target_parameters))
            d_ff = round_to_multiple(8 * hidden / 3, 128)
            candidates.append((error, hidden, layers, d_ff))
    return min(candidates)
```

为什么用 log error？把目标放大 20% 与缩小 20% 应受到近似对称惩罚；直接 absolute parameter error 会天然偏向小模型。

### 2.3 run 状态与 validation loss

训练被切成 `n_evals` 个 chunks，每块训练 `total_train_tokens/n_evals` tokens，然后验证，loss 序列位于 `status.val_losses`。只有 `status_type == "completed"` 时，最后一个元素才是该 run 的 final validation loss。timeout/failed 可能带 partial losses，可用于诊断学习率或粗看吞吐，但不能冒充完成训练的 final loss 放进同一拟合。

数据顺序固定、无 epoching，最多使用 500B tokenized DCLM tokens；同一配置消费相同的确定性顺序。`model_seed` 只控制模型初始化，不改变 data order。验证集为约 $2^{18}=262,144$ tokens；final leaderboard 会使用更多 validation tokens，因此小验证集上的微小差距也含 sampling noise。

## 3. 用 12 小时买到最有信息的 runs

### 3.1 先写实验表，再发请求

一个保守的最大预留方案如下；完成得早会返还预算，但决策时不要依赖返还：

| 阶段 | 数量 × max runtime | 上限 | 要回答的问题 |
|---|---:|---:|---|
| API/吞吐 pilots | 6 × 120s | 0.20h | config 合法吗？loss 会降吗？每种 shape 的 tokens/s？ |
| 四条 IsoFLOPs profiles | 5 × (300+600+1200+2400)s | 6.25h | 每个 C 的 U 形最低点在哪里？ |
| LR/batch 局部 sweep | 8 × 600s | 1.33h | 默认优化器是否系统性压低某些模型？ |
| held-out + seed repeats | 4 × 900s | 1.00h | 外推能预测未参与拟合的 budget 吗？关键点噪声多大？ |
| 故障与边界扩展储备 | — | 3.22h | timeout、边界最低点、补测 |

每个候选 run 提交前写一句“这个 run 会区分哪两个决策”。若答案只是“多一个点可能有用”，先不要花预算。

### 3.2 怎样构造每条 profile

1. 用 pilots 测不同 shape 的有效训练速率：`effective_flops_per_second = 6*N*D/used_runtime_seconds`；
2. 选几何递增的 C 档，而不是在线性尺度平均撒点；
3. 对每档围绕预估 Nopt 取乘数 `[0.4, 0.65, 1.0, 1.55, 2.5]`；
4. 对每个 N 用 `D=C/(6N)`，再向下对齐 `512*batch`；
5. 固定 optimizer、data、seed，避免把学习率变化误认为 scaling；
6. 若最低 loss 位于最小/最大 N，说明尚未包住谷底，必须用储备向外扩一格。

### 3.3 throughput 不能只用一个常数

最终预算是 48 B200 wall-clock hours，而 scaling 公式使用 FLOPs。先用 completed runs 估计不同 shape 的 effective FLOP/s，至少对 `log throughput` 回归 `log N`、depth/width ratio、batch、dtype 等变量，或在最终候选附近用局部实测。然后把 48 小时换为候选 shape 实际能执行的 C/D。

如果预测 `Dopt` 在 48 小时内无法处理，就不能仍提交它；应在时间可行集合上重新最小化 predicted loss。反之若 run 提前处理完 Dopt，额外 token 可能继续降低 data-limited term，需比较更大 D 的候选。

## 4. 两种拟合互相校验

### 4.1 方法 A：IsoFLOPs minima → 两条 power laws

这就是第 1 节的方法。优点是直观、参数少；缺点是每档只留下一个最低点，丢掉其他 run 信息，而且网格粗糙会让 minima 跳动。报告每条 profile 的原始点、最低点是否内部、log residual 与 held-out 预测误差。

### 4.2 方法 B：拟合完整 Chinchilla loss surface

常见形式：

$$L(N,D)=E+\frac{A}{N^\alpha}+\frac{B}{D^\beta}.$$

$E,A,B,\alpha,\beta$ 都应为正。配套 [`scaling_fit.py`](https://github.com/Palind-Rome/cs336-learning-journey/blob/main/examples/assignment3/scaling_fit.py) 在 log-parameterization 下用 `scipy.optimize.least_squares` 和 soft-L1 robust loss：

```python
def decode(theta):
    E, A, alpha, B, beta = np.exp(theta)
    return ChinchillaLaw(E, A, alpha, B, beta)

def residual(theta):
    law = decode(theta)
    return (law.predict(N, D) - losses) / losses

result = least_squares(residual, initial, loss='soft_l1', f_scale=0.02)
```

用 relative residual 是为了不让少数高 loss pilots 因绝对数值更大而支配拟合。固定 C 后令 $D=C/(6N)$，在 log N 上做 bounded scalar minimization即可得到 Nopt/Dopt/predicted loss。

不应只报告 optimizer 的一个点估计。对 completed runs 做 bootstrap，每次有放回重采样、重新拟合并重新求最优，报告 Nopt/Dopt/loss 的 5/50/95 percentiles。如果两种拟合方法给出相差很大的最优配置，这本身就是重要证据：应把剩余预算花在二者分歧最大的区域，而不是草率平均。

### 4.3 held-out 检查

至少留出一整档 compute 不参与拟合。用 scaling law 预测它的 Nopt 与 loss，再看真实 profile；报告：

- predicted/observed Nopt ratio；
- predicted loss 与 observed minimum 差；
- 残差是否随 N 或 C 呈系统趋势；
- seed repeat 的差异是否大于候选间差异。

若 residual 有明显曲线，幂律范围可能过宽；若大模型一侧系统偏差，learning rate 或 runtime/throughput 可能没有正确随规模调整。

## 5. 从连续 Nopt 落到最终 config

scaling law 给的是连续参数量，不会自动告诉你 layers、width、heads、batch 与 learning rate。按以下顺序固定：

1. 用 `nearest_architecture(Nopt)` 生成目标附近 2–3 个合法 shapes；
2. 用源码真实统计参数量，不能只依赖 $12Ld^2$ 近似；
3. head_dim 先保持 64，保证 heads×head_dim=hidden；
4. `intermediate_size` 取接近 $8d/3$ 的 128 倍数；
5. 在小规模 runs 中拟合/观察最佳 peak LR 随 N 或 batch 的变化；
6. batch 要兼顾吞吐、steps 数与 token divisibility，不因 batch 变大就忘记调整 LR；
7. 根据局部 throughput 选择 48 小时内可完成的 `total_train_tokens`；
8. 用 loss surface 预测 final loss，并带上 bootstrap interval；
9. 在真正 `POST /final_submission` 前保存完整 JSON、拟合代码 commit 与图表。
