第一阶段解决“算什么”，这一阶段解决“**在哪里算、数据怎样到达计算单元、什么时候通信**”。系统优化的正确性底线是：换了 kernel 或并行策略，模型仍应得到与单卡参考实现等价的结果。

```text
数学图：matmul → norm → attention → optimizer
             ↓ 映射到硬件
执行图：HBM ↔ SRAM/register ↔ Tensor Core ↔ network
```

<ConceptBridge title="先把 GPU 想成一座吞吐工厂">
  Tensor Core 是速度极快的生产线，HBM 是远处仓库，片上 SRAM/register 是工位旁的小料架。若每做一点计算都回仓库拿材料，生产线再快也会闲着；tiling、fusion 与 FlashAttention 的共同目标，是让搬来的一块数据在片上多做一些有用工作。
</ConceptBridge>

## 进入性能代码前先分清四个指标

| 指标 | 回答的问题 | 常见误用 |
|---|---|---|
| latency | 单次操作等多久 | 把异步 launch 时间当完整 GPU 时间 |
| throughput | 单位时间完成多少 token/样本 | 忽略 batch 和 sequence 配置 |
| memory | 峰值与常驻状态各占多少 | 只看参数，不看 activation/optimizer |
| communication | 传多少 bytes、是否阻塞 | 只数 collective 次数，不看拓扑与 overlap |

## 每次优化都走同一个闭环

```text
写数值 oracle
  → 选固定 shape/dtype/device
  → warmup + synchronize + benchmark
  → profiler 定位瓶颈
  → 只改一个资源路径
  → 对齐 forward / backward / distributed result
  → 重新测量并记录适用范围
```
