## Lecture 12：评测是测量学，不是排行榜

到这里已经有 architecture、训练循环、分布式系统与 scaling recipe；下一步看似是“找数据”，但数据选择必须先回答一个更根本的问题：希望模型产生什么行为，怎样知道它真的产生了？Evaluation 的核心映射是：

$$
\underbrace{\text{能力、帮助性、安全性等抽象构念}}_{\text{construct}}
\quad\longrightarrow\quad
\underbrace{\text{可执行 protocol 与数字}}_{\text{measurement}}.
$$

排行榜只展示右侧。若这条映射不成立，分数算得再精确，也只是精确测错了东西。

### 1. 先写“测量声明”，再写 prompts

一个可审计 evaluation spec 至少包含：

1. **对象**：base LM、chat model、固定 decoding 的 system，还是带工具的 agent？
2. **目标构念**：知识、推理、写作偏好、真实任务完成率、特定风险，不能只写“总体能力”。
3. **实例分布**：谁的请求、哪个时间段、哪些语言/领域，怎样抽样。
4. **交互协议**：system prompt、few-shot、工具、token/time/cost budget、是否允许重试。
5. **评分规则**：exact match、log probability、unit tests、pairwise judge 或人工 rubric；invalid output 怎样算。
6. **不确定性**：样本误差、decoding 随机性、judge variance 与多次运行。

“给一组 prompt → 取 response → 算 accuracy”只覆盖最后几步。真正决定结论的是：题目是否代表目标、grader 是否漏判、实验是否给不同系统同等资源，以及数字是否足以区分两个相近模型。

### 2. Perplexity：最干净，也最容易比较错

对 token 序列 $x_{1:T}$，平均 negative log-likelihood 与 perplexity 是：

$$
\operatorname{NLL}(x)=-\frac1T\sum_{t=1}^{T}\log p(x_t\mid x_{<t}),
\qquad \operatorname{PPL}(x)=e^{\operatorname{NLL}(x)}.
$$

PPL 可解释为模型在每一步面对的等效分支数；它对所有 token 使用 strictly proper scoring rule，不能靠只给一个答案标签作弊。经典 in-distribution 设置在 Penn Treebank、WikiText-103、One Billion Word 的 train/test split 上训练和测试；GPT-2 则在 WebText 训练，再 zero-shot 测这些 OOD corpus，展示 transfer 在小数据域可能更有利。

但原始 PPL 回答的是“整个字符串在模型分布下多意外”，未必是任务关心的部分。给 prompt $c$ 与 response $y_{1:m}$，conditional NLL 应只累计 response：

$$
-\frac1m\sum_{i=1}^{m}\log p(y_i\mid c,y_{<i}).
$$

实现时 label mask 把 prompt positions 设为 ignore index；若把 prompt 也除进分母，长 prompt 会把差异稀释。还要明确 BOS/EOS、换行、前导空格、最大长度与 sliding-window 边界。

不同 tokenizer 下 token 数 $T$ 不同，token-PPL 不可直接横比。可报告总 log-likelihood 并按原始 bytes/characters 归一化，或只在相同 tokenizer/序列化协议内比较。Cloze 的 LAMBADA 与 sentence completion 的 HellaSwag，本质上也可用候选 continuation 的 conditional log probability 排名。

还有一个容易忽略的安全问题：若提交者只交一个声称是概率的 API，leaderboard 必须验证 distribution 是否归一化；否则任意给正确 token 超过 1 的“伪概率”可获得不可能的 log score。下游 generation+accuracy 不需要信任完整概率表，但会引入 decoder 与 parser 误差。

### 3. 考试 benchmark：题目明确不代表 protocol 唯一

MMLU 有 57 个学科，主要测知识而非名称所说的广义 language understanding；原 GPT-3 结果使用 few-shot。MMLU-Pro 清除噪声/过易题，把 4 个选项扩成 10 个，并用 chain-of-thought，模型准确率下降约 16–33 个百分点。GPQA 由 61 位 PhD contractor 编题：领域专家约 65%，非专家有 Google 且用 30 分钟约 34%，当时 GPT-4 约 39%。HLE 汇集 2,500 道多模态、多学科选择/短答题，经 frontier models 过滤、多轮审核，并用 50 万美元奖金与共同作者身份激励出题。

同一选择题至少有三种 scorer：

~~~python
def score_choice(model, prefix: str, choices: list[str]) -> int:
    # 每个候选必须使用完全相同的 prefix 与边界序列化。
    scores = []
    for choice in choices:
        token_ids = model.tokenize(" " + choice)
        logp = model.conditional_logprobs(prefix, token_ids)
        scores.append(sum(logp))  # 或显式声明 length-normalized score
    return max(range(len(scores)), key=scores.__getitem__)
~~~

- 生成“A/B/C/D”再 parse，接近 instruction-following，却会把格式遵循计入能力。
- 比选项字母的 next-token logprob，便宜但受字母/顺序先验影响。
- 比完整选项 continuation logprob，较语义化，但长短 normalization 会改变答案。

因此要固定 option order 或做 permutation test，声明 zero/few-shot、CoT、sampling、答案抽取与 invalid handling。题变难可以延缓 saturation，却仍不自动增加生态效度：真实用户通常不问有唯一标准答案的十选一问题。

### 4. 开放回答：pairwise preference 测到哪些混合信号

Chatbot Arena 让真实互联网用户输入 prompt，匿名展示两个随机模型的 response，再选择较好者。二元结果可用 Bradley–Terry/Elo 形式：

$$
P(A\succ B)=\frac{1}{1+10^{(r_B-r_A)/400}}.
$$

拟合的是在 **Arena 当期用户与 prompt 分布** 上的相对偏好，不是客观真理。它的优点是请求真实、无需所有模型回答完全相同 prompts、可动态加入新模型；限制是用户身份/动机未知，spam 与地域语言分布不可控，偏好把 correctness、style、verbosity、sycophancy 和 brand impression 混在一起，人也未必有能力核对答案。

AlpacaEval 用 805 条 instruction，让 GPT-4 preview 判断候选相对 GPT-4 preview baseline 的胜率；早期 LLM judge 偏爱长答案，导致 leaderboard gaming，2.0 用回归作 length-controlled debiasing。WildBench 从 100 万 human-chatbot conversations 中取 1,024 例，让 GPT-4 Turbo 先按 checklist 再判断，与 Arena 的相关性作为 sanity check。

稳健 judge protocol 应：匿名模型名；随机交换 A/B；允许 tie；把 correctness、relevance、style 拆成 rubric；测同一对 response 复判一致率；使用多个 judge；用人工 gold subset 估计 judge error；同时报告 raw 与 length/style-controlled score。Arena correlation 只能说明两个 metric 在现有模型集上排序相近，不能证明新模型或新语言上仍有效。

### 5. Agent benchmark：一次回答变成一条状态轨迹

agent 是 LM 加 scaffold。SWE-bench 给 codebase 与 issue description，要求提交 patch，以 unit tests 判断；原版有 12 个 Python repositories、2,294 个 tasks。Terminal-Bench 用通用 computer terminal 环境，2.0 从 93 位贡献者众包的 229 个 tasks 中精选 89 个。CyBench 有 40 个 CTF tasks，并用人类 first-solve time 表示难度。MLE-Bench 把 75 场 Kaggle competitions 变成需要处理数据、训练模型与提交结果的任务。

此时最小评测单位不再是 response，而是：

$$
(\text{model},\text{scaffold},\text{tools},\text{environment image},
\text{budget},\text{random seed})\to\text{trajectory}\to\text{outcome}.
$$

显式 todo、层级委派、持久文件 memory 和更强 context engineering 都能显著改变得分。必须冻结 container digest、repository commit、网络权限、tool versions、wall-clock/token/cost limit 与 retry policy；同时保存动作轨迹。只报“某模型 SWE-bench 60%”而不报 scaffold，无法复现，也不能归因于 LM 本身。

Unit tests 是可扩展 verifier，但 coverage 不足会让错误 patch 过关，甚至 trivial agent 也能利用 benchmark 缺陷。SWE-bench Verified、Platinum benchmark 与 Docent 式 trace inspection 都在修复“grader 错而非模型错”的问题。对随机 agent 可报告 pass@1、多个独立 seeds 的 success rate 与成本分布；挑最好一次会系统性高估线上表现。

### 6. 推理、安全与现实性不是同一根轴

ARC-AGI 试图把 reasoning 与 world knowledge 分开：每题新颖、对人可解，记忆帮助较小。ARC-AGI-1 是 2019 的彩色网格任务；2025 的第二版加入更多多步推理；2026 的第三版变成 interactive environments。预训练 LM 对早期版本提升有限，o1/o3 类 reasoning model 才明显上升。它暴露当前模型缺口，但仍限定于人类设计、对人可解的 reasoning，不等于测“所有智能”。

Safety 也不是一个标量。HarmBench 以 510 个违法或违背规范的 harmful behaviors 测拒绝/越狱；AIR-Bench 从 regulatory frameworks 与 company policies 建 taxonomy，覆盖 314 个风险类别、5,694 prompts。GCG 通过 greedy coordinate gradient 自动优化 adversarial suffix，并能从 Llama 等开放权重模型迁移到 GPT-4 等闭源模型。Hallucination、sycophancy、协助犯罪、不平等与削弱批判思维的 mechanism/受害方都不同；政治、法律与社会规范还随地区变化。Cyber agent 更体现 dual use：同一能力既可入侵，也可做授权 penetration test。

Ecological validity 问的是任务是否像真实使用。GPQA 与标准考试离实际工作很远；Arena prompts 来自真实人群，却分布不可控。GDPVal 从美国 GDP 最大的九个 sectors 选 44 种 occupations，任务由平均约 14 年经验的 professionals 提供。MedHELM 不再只依赖医学标准考试，而由 29 位 clinicians 提供 121 个临床任务，混合 private/public datasets。Anthropic Clio 用 LM 分析真实用户数据，只分享聚合使用模式。越真实的数据通常越涉及隐私，现实性与可公开复现性存在结构性张力。

### 7. Validity：污染、坏题与错 grader 都会伪造进步

foundation model 在互联网训练，而 providers 往往不公开 data manifest，传统固定 train/test split 被打破。污染检查有四条路线：

- 从模型行为推断 overlap，例如利用数据点 exchangeability，但这是间接统计证据。
- 建立报告规范，让 provider 报 overlap 与 confidence intervals。
- 使用 fresh eval，如 LiveCodeBench/UncheatableEval 抓取训练 cutoff 后的新网页；但 timestamp 仍可能因复制而失真。
- 使用 private eval，如企业内部 codebase 或个人未公开 writings；最容易用于 perplexity，却不利公开复现。

数据侧还可对已知训练 corpus 做 exact hash、n-gram/MinHash 近重复检索和时间切分；canary 能测特定字符串记忆。模型复述一道题不是污染的充分条件，因为公开知识可能独立出现；不能复述也不是无污染证明，因为训练优化并不保证逐字记忆。

benchmark 本身也需 versioning 与 QA。SWE-bench Verified 修正不可解/不可靠任务，Platinum 版本试图提高题与 grader 质量；agent benchmark 可能 test cases 不足或可被 trivial shortcut 解决。每次升级都应保留旧版本、变更日志、instance IDs 与 paired results，否则“模型提升”可能只是评测集换了。

### 8. 用途决定规则，不存在唯一总分

购买者想在自己的客服流量上选 A/B；研究者想隔离 raw capability；企业和政策制定者想理解 benefits/harms；开发者想得到可行动的 failure feedback。这四个目的需要不同样本与 metric。早期 ImageNet/SQuAD 主要比较固定数据和预算下的 **methods**；今天多数榜单比较允许任意训练数据与系统优化的 **models/systems**。nanoGPT speedrun 是仍在比较 method 的例外：固定数据，以到达指定 validation loss 的 compute time 为目标。

一份最终 evaluation card 应把规则、数据版本、prompt template、decoder、工具、grader、成本与置信区间一起发布，并给出 instance-level failure slices。难度、现实性与效度是三个独立问题；“题很难”不保证真实，“来自真实用户”也不保证 grader 正确。

以下对应 <code>lecture_12.py</code> 的全部讲授文本、示例与外链；图片所展示的榜单、流程和题例在相邻句中说明。

**开场与问题定义。** 第 12 讲主题是 evaluation。此前已讲完训练 LM 所需的 architecture、training、systems 与 scaling；缺少的是用什么 data 训练。数据会塑造模型行为，例如代码、多语言或 DNA 能力。但讨论数据之前，必须先说明希望模型呈现何种行为。Evaluation 问：给定模型，它有多“好”？

表面看评测很机械：一，定义 prompts；二，把 prompts 发给模型得到 responses；三，计算 accuracy。事实上评测深刻而重要，并会塑造 AI 的发展。核心挑战是把红色的抽象 construct 映射到蓝色的 concrete metric。也许“好”是 Artificial Analysis 上 benchmark 高；也许还要便宜；也许是 Arena 上用户更偏好；也许是用户愿意使用并在 OpenRouter 上付费。讲义展示对应的 benchmark、cost、Arena leaderboard 与 OpenRouter usage 图。

**Perplexity。** 语言模型是 token sequence 上的概率分布 $p(x)$；$(1/p(D))^{1/|D|}$ 衡量 p 是否给 dataset D 高概率。Pretraining 最小化 train-set perplexity，最自然的做法是在 test set 测 PPL，这也是传统 LM research 范式。标准数据有来自 Wall Street Journal 的 Penn Treebank、Wikipedia 的 WikiText-103，以及由 WMT11 机器翻译材料 EuroParl/UN/news 构成的 One Billion Word Benchmark。经典范式是在同一 dataset 的 train split 训练、test split 测试，即 in-distribution evaluation；纯 CNN+LSTM 在 1BW 把 PPL 从 51.3 降到 30.0。

GPT-2 在 40GB WebText——Reddit 外链网页——上训练，再对标准数据 zero-shot，属于 out-of-distribution evaluation。图表显示 transfer 对 PTB 等小数据集帮助更大，对 1BW 等大数据集则没有同样优势。

“Perplexity is all you need”更像信念而非科学：若真实分布为 t、模型为 p，理论最佳 perplexity 是 $H(t)$，当且仅当 $p=t$；若 p 等于 t，就能通过 $p(\text{solution}\mid\text{problem})$ 解所有任务，于是不断降低 PPL 最终会“到达 AGI”。反过来，PPL 也许测得过多：在“Stanford was founded in 1885”中，它惩罚所有 token，包括可能与目标知识无关的 founded；解法是只测 $p(\text{response}\mid\text{prompt})^{1/|\text{response}|}$ 的 conditional PPL。

一些 benchmark 是伪装的 perplexity：LAMBADA 是 cloze/fill-in-the-blank，HellaSwag 是 multiple-choice sentence completion，讲义展示各自样例。若运行 PPL leaderboard，用户提交 LM、平台计算 <code>log_prob = LM(test_data)</code>，就必须相信其 probabilities 有效且和为 1；下游任务则取 <code>response = LM(prompt)</code> 再算 response accuracy。总结：PPL 因曲线平滑，仍大量用于 LM development；不信“PPL 足够”的人还需要捕捉真实情境的 benchmarks。

**考试评测。** 像测试人一样，考试可控制 subject/difficulty，设计无歧义答案且容易评分。MMLU 有 57 个科目，如 math、US history、law、morality，是 graduate/undergraduate students 从网上免费来源收集的 multiple-choice；尽管名称如此，它主要测 knowledge 而非 language understanding，并在 GPT-3 上用 few-shot prompting。讲义展示题目，并链接 llm-stats 与 HELM 的逐预测可视化。

MMLU-Pro 删除 MMLU 的 noisy/trivial questions，把 4 choices 扩到 10 choices，并用 chain of thought 给模型更多机会；模型 accuracy 下降 16%–33%，饱和度更低。讲义同样链接统计与 HELM leaderboard。

GPQA 全称 Graduate-Level Google-Proof Q&A，由从 Upwork 雇佣的 61 位 PhD contractors 编写。PhD experts 得分 65%；non-experts 可用 Google、每题 30 分钟仍只有 34%；当时 GPT-4 是 39%。讲义展示示例并链接结果。

HLE 全称 Humanity’s Last Exam，共 2,500 道 multimodal、多学科、multiple-choice 与 short-answer questions；向 creators 提供 50 万美元 prize pool 与 co-authorship，经 frontier LMs 过滤并进行多阶段 review。讲义展示题例、筛选 pipeline、结果与排行榜。

考试趋势是随着模型饱和旧 benchmark，题目越来越难；multiple-choice 可被做得任意难。但它不捕捉真实使用：现实请求是 open-ended，也未必存在唯一正确答案。

**聊天评测。** 目前任务都有明确 multiple-choice，但多数人不会向 AI assistant 提选择题。例子是：“我想做甜菜配山羊奶酪沙拉，哪些香草合适、哪些不合适？”回答需要按甜菜的甜土味与山羊奶酪的酸香奶油感分析搭配。挑战是怎样评价 open-ended response。

Chatbot Arena 的数据收集流程是：互联网随机用户输入 prompt；两个随机且匿名的 models 返回 responses；用户选择较好者。用 pairwise comparisons 计算 Elo，模型为 $p(A\text{ beats }B)=1/(1+10^{(ELO_B-ELO_A)/400})$，再最大化已观察胜负的概率。它拥有真实 prompts，免费使用给用户真实使用动机；但用户是谁、有什么 bias、是否 spam 并不清楚。Binary preference 混合 style/correctness，人类未必能评估 correctness，也可能偏好 sycophancy。优点是无需向所有模型发送相同 prompts；系统会随新 prompts/models 动态变化。

AlpacaEval 2023 有来自多源的 805 instructions；metric 是相对 GPT-4 preview baseline 的 win rate，judge 也用 GPT-4 preview，因此可能有 bias。LLM judge 偏好更长 response，曾导致 leaderboard gaming；AlpacaEval 2.0 用 regression 去偏。怎样评估这个 metric？它与使用 humans 的 Chatbot Arena 高相关，讲义给出相关图与 leaderboard。

WildBench 从 100 万条 human-chatbot conversations 取 1,024 examples；用 GPT-4 Turbo 依据 checklist——像 judge 的 CoT——再做评判。它也与 Chatbot Arena 高相关，后者似乎成了事实上的 sanity check；讲义链接 HELM 可视化。总结：open-ended response 难评；相近 answers 的 pairwise comparison 信号更强；human 与 LLM judge 都有 bias；无论谁评分，checklist/rubric 都能提高 reliability。

**Agent 评测。** 之前评价 LM 说什么，现在评价 LM 做什么。Agent 等于 language model 加决定怎样调用 LM 的 scaffold；任务需要 tool use，例如运行代码，并在一段时间内迭代。

SWE-bench 有 12 个 Python repositories 的 2,294 tasks；给 codebase 与 issue description，要求提交 PR，metric 是 unit tests。Terminal-Bench 使用简单、通用的 computer terminal environments；229 tasks 来自 93 contributors，精选 89 个组成 Terminal-Bench 2.0，讲义展示 human completion time 与结果。CyBench 包含 40 个 Capture the Flag tasks，用 first-solve time 衡量 difficulty，图示 agent loop 与结果。MLE-Bench 有 75 个 Kaggle competitions，需要训练模型、处理数据等，讲义展示任务与成绩。

Agent scaffold 可以显式 planning——维护并勾选 todo；hierarchical delegation——agent 调 sub-agent 以获得干净 context；persistent memory——读写文件；以及 extreme context engineering——明确给更多 process instructions。总结：agents 大幅扩展 LM capability surface；scaffold 极重要；评估 agent 等于同时评估 scaffold 与 LM。

**纯推理评测。** 前述任务都需要语言与 world knowledge。能否从 knowledge 中隔离 reasoning？Reasoning 也许可代表更纯的 intelligence，而非只记事实。ARC-AGI 的目标是人类 100% 可解但 AI 困难；每题独特，memorization 无帮助。ARC-AGI-1 是 2019 首版彩色网格；ARC-AGI-2 于 2025 年 3 月加入更多 multi-step reasoning。结果显示 pretrained LMs 没有明显推进，o1/o3 等 reasoning models 才让曲线上升。ARC-AGI-3 于 2026 年 3 月改为 interactive environments，讲义展示环境与结果。总结：目标是分离 reasoning/knowledge，虽然很难；它受限于 human reasoning，而不是 superhuman reasoning；却能清晰暴露当前模型缺口。

**安全评测。** 讲义先用汽车碰撞测试图追问 AI safety 是什么。HarmBench 基于 510 个违反 laws/norms 的 harmful behaviors，并链接 HELM leaderboard 与一个具体 failure。AIR-Bench 从 regulatory frameworks 与 company policies 出发，整理为 314 个 risk categories、5,694 prompts，讲义展示 taxonomy 并链接 HELM。

Jailbreaking 的背景是 LM 被训练为拒绝 harmful instructions；Greedy Coordinate Gradient（GCG）自动优化 prompts 绕过 safety，并能从 Llama 等 open-weight models 迁移到 GPT-4 等 closed models，讲义展示样例。Safety 的许多方面高度 contextual：politics、law、social norms 随国家变化；风险也很多样，包括 hallucination、sycophancy、协助犯罪、inequality、失去 critical thinking。Dual-use 的例子是强 cybersecurity agent Mythos，既可用于入侵，也可做 penetration testing。

**现实性。** Ecological validity 指 evaluation 捕捉 real-world use 的程度。GPQA 等考试离真实使用很远；Arena prompt 来自真实人，但 distribution 不受控。OpenAI GDPVal 选美国 GDP 前九 sectors 的 44 occupations，tasks 来自平均约 14 年经验的 professionals。MedHELM 针对旧医学 benchmark 依赖 standardized exams 的问题，由 29 clinicians 提供 121 clinical tasks，混合 private/public datasets。Anthropic Clio 用 LM 分析 real user data，并分享用户在问什么的总体 pattern。遗憾的是 realism 与 privacy 有时互相冲突。

**效度与 train–test overlap。** ML 基础原则是不能在 test set 训练。ImageNet/SQuAD 等 pre-foundation model 时代有明确 split；今天模型训练互联网，却不公开 data。路线一是从模型推断 overlap，例如利用 data points 的 exchangeability；路线二是鼓励 reporting norms，让 providers 报 overlap 与 confidence intervals；路线三是 fresh eval，LiveCodeBench/UncheatableEval 抓新网页，但 timestamp 也会因复制失效；路线四是 private eval，用企业内部 codebase 或个人 writings，对 PPL 尤其容易。

Dataset quality 同样重要：SWE-bench 被修成 SWE-bench Verified；有人建立 benchmark 的 Platinum versions；agentic benchmark 可能 test cases 不足，甚至 trivial agent 能解；Docent 用 LLM 检查 agent traces 以发现这些问题，讲义逐一链接相关工作与修复前后示例。

**怎样思考 evaluation。** 不存在唯一正确评测，它取决于问题。一，用户/公司为客服 chatbot 等场景在 model A/B 间做购买决定；二，研究者测模型 raw capabilities，如 intelligence；三，出于 business/policy 理解 benefits 与 harms；四，开发者获取改进反馈。

Pre-foundation models 时代评估的是固定 train/test split 下的 methods；今天主要评估允许任意手段的 models/systems。例外是 nanoGPT speedrun：固定 data，以到达指定 validation loss 的 compute time 为目标。评 methods 鼓励研究者做 algorithmic innovation；评 models/systems 对 downstream users 有用。无论哪种，都必须定义游戏规则。

**最终要点。** 没有唯一真实 evaluation，应按想测的对象选择；必须清楚声明比较 methods、models 还是 agents 的规则；并同时考虑 difficulty、realism 与 validity。

### 1. 训练阶段也是数据质量的漏斗

典型流程由 pre-training、mid-training、post-training 组成。Pre-training 吃海量网页/文档，建立语言与世界模型；mid-training 用较少、较高质量或能力定向数据继续训练；post-training 使用对话、偏好与 reinforcement learning signals 改变交互行为。边界并不严格，但常见趋势是数量下降、单 token 成本与目标性上升。

“Base model”在本讲口径中是 pre+mid-training 后的模型，“instruct/chat model”是 post-training 后的系统；越来越多机构只发布 instruct checkpoint。OLMo 2 的公开材料把 pretraining、Dolmino mid-training 与 Tülu post-training 分开，正好说明每阶段需要独立 data manifest，而不能只说“总共训练了 X tokens”。

数据比 architecture 更像 long-tail operations：一个规则可处理 99% 页面，余下 1% 仍可能是数十亿个 PDF、乱码、模板、隐私、许可与 adversarial cases。Architecture code 可被完整发布，训练 data 却常因竞争优势与版权风险仅披露几个比例。

### 2. 从 live server 到 immutable snapshot

Crawler 从数亿 seed URLs 维护 frontier：取 URL、尊重访问策略、下载 HTTP response、解析 links、规范化 URL、判重并继续。它需要三类 policy：selection 决定先抓什么；politeness 读取 robots.txt、限制并发/频率，避免拖垮站点；revisit 决定多久重抓变化页面。

动态 app 的 URL 不随点击改变，Discord、Weights & Biases 等内容要交互才能出现；Facebook、X、LinkedIn、NYTimes 等还需登录/付费。Cloudflare/CAPTCHA、IP/国家封锁与 rate limit 是技术边界；robots.txt、ToS、license 与适用法律是规范边界。Shadow libraries 虽能从 web 访问，却通过绕 paywall/版权控制聚合 LibGen、Z-Library、Anna’s Archive、Sci-Hub 等材料；“可 curl 到”绝不等于有合法来源权利。

Common Crawl 是 2007 年成立的非营利组织，约每月抓取一次并发布可下载 snapshot。WARC 保留 raw HTTP，WET 是已经抽成 text 的有损版本。选择 WET 很省工程，却把 trafilatura/resiliparse/jusText 等 extractor 的决定永久固化；DCLM 实验表明仅 HTML→text 工具差异就会改变 downstream accuracy。可复现实验应记录 crawl ID、segment、WARC record locator 与 extractor version。

### 3. 权利栈：不要把五个问题压成一个“能不能用”

以下是课程内容的工程解读，不替代具体司法辖区的法律意见。至少要分别回答：

| 层 | 问题 | 典型证据 |
| --- | --- | --- |
| Access | 技术上能否取得，是否绕过认证/控制 | fetch log、robots snapshot、auth method |
| Contract | 使用者是否受 ToS/API agreement 约束 | terms version、account/contract |
| Copyright | 复制、改编、分发涉及哪些权利 | author/source、publication、registration、jurisdiction |
| License/exception | 获得了什么许可，或是否主张 fair use | SPDX/CC text、license provenance、用途分析 |
| Privacy/safety | 是否含 PII、secret、未同意对话或危险内容 | consent basis、PII scan、access control、deletion process |

版权保护 expression 而非 abstract idea；门槛低且通常无需先注册才产生。License 是权利人允许特定行为的合同授权，Creative Commons、MIT、Apache 等各自带 attribution/share-alike/noncommercial/patent 等不同条件。“dataset 整体采用 ODC-By”也不自动清洗其中每份作品的权利，这叫 collection-level license 与 item-level rights 的差别。

美国 Copyright Act §107 要逐案权衡 purpose/character、作品 nature、使用 amount/substantiality 与对潜在市场的 effect；没有“少于 X% 就自动合理使用”的公式。Google Books 的索引/片段展示是经典案例，但它不能机械推出 foundation-model training 的所有事实组合。讲义列举的 NYT v. OpenAI、Bartz v. Anthropic、Kadrey v. Meta 也区分训练用途、取得副本的方式与输出/市场事实；网站只保留讲义截至编写时的案件梳理，并链接 [美国版权局 §107 原文](https://www.copyright.gov/title17/92chap1.html#107) 与 [Fair Use 说明](https://www.copyright.gov/fair-use/more-info.html)。

工程上应建立 rights manifest：每个 source 的取得方式、license 表达式/证据、允许用途、attribution、地域/时间、opt-out/deletion key 与下游 redistributability。训练许可、发布 processed dataset、发布 examples、发布 weights 是不同动作，不能用一列 boolean 代替。

### 4. 专门数据源：结构价值与偏差一起进入模型

| Source | 可获得结构 | 主要价值 | 主要风险/偏差 |
| --- | --- | --- | --- |
| Wikipedia dumps | article、revision、language、links | 清洁百科知识、多语言、定期 dump | notability/编辑者偏差、文体单一、dump 前 poisoning |
| Git repositories | tree、commits、diffs | code、长期变化、tests | forks/复制、generated/vendor、secrets、复杂 license |
| GitHub metadata | issues、PR、review、votes | 问题→修改→讨论的真实过程 | bot/spam、隐私、API/ToS、与 repo 版本对齐 |
| arXiv | metadata、PDF、可选 TeX | 论文、公式、研究语言 | 非 peer review、rights 由作者选择、PDF 抽取损失 |
| Stack Exchange | Q/A、score、comments、tags | 接近 instruction/answer，metadata 可筛选 | 人群/主题偏差、许可与 attribution |
| Books | 长程叙事与连贯文档 | long-context、文体、知识 | 获得方式和版权差异巨大 |

Wikipedia dump 避免自己 crawl，但“任何人可编辑”也产生 dump-time poisoning 窗口。Git 应直接 clone repository，而不是抓 HTML；GitHub Archive 等提供 event/metadata，Software Heritage 聚合多 host 的 source history。arXiv metadata/abstract 是 CC0，PDF/TeX 的许可由作者选择，不能因从同一 S3 bucket 下载就统一处理。Document boundaries 也重要：BERT 明确按 documents 而非零散 sentences 训练，以保留长依赖。

### 5. 2018–2023：从单一来源到可控 mixture

数据谱系不是“后者替代前者”，而是不断换质量代理：

- **BERT**：Wikipedia + BooksCorpus。BooksCorpus 从 Smashwords 抓约 7K 本定价为 0 的自出版书、985M words，后因违反 ToS 下线。
- **GPT-2 WebText**：把 Reddit 帖子中 karma≥3 的外链当人工质量 vote，得 8M pages/40GB；OpenWebText 从 Reddit submission URLs 复刻，并做 fastText English filter 与 near-dedup。
- **CCNet**：对 Common Crawl 做 paragraph dedup、fastText language ID，再用 Wikipedia 训练的 KenLM 5-gram PPL 作质量相似度；目标也包括 Urdu 等低资源语言。
- **C4/T5**：从 2019-04 Common Crawl 1.4T tokens 出发，用行尾标点、≥5 words、≥3 sentences、bad-word、代码/placeholder/terms-of-use 与高置信 English 等规则，剩 806GB/156B tokens。
- **GPT-3**：processed Common Crawl、WebText2、Books1/2、Wikipedia，共约 570GB/400B tokens；classifier 把 WebText/Wikipedia/books 当 positives，并 fuzzy-dedup 文档与 benchmarks。
- **The Pile**：开源社区聚合 22 个 domain、825GB/约275B tokens；Pile-CC 从 WARC+jusText 重抽，另含 PubMed Central、arXiv TeX、Enron emails、Project Gutenberg、Books3、StackExchange 等。
- **MassiveText/Gopher**：English、dedup、benchmark overlap、manual quality rules 与 Google SafeSearch toxicity；共称 10.5TB，但 Gopher 只采约 300B tokens/12%。
- **LLaMA**：CCNet Common Crawl、C4、permissive GitHub、20-language Wikipedia、Gutenberg/Books3、清理过的 arXiv 与按 score 排序的 28 个 StackExchange sites，共 1.2T tokens；RedPajama 复刻，SlimPajama 用 MinHashLSH 去重成 627B。
- **RefinedWeb/Falcon**：直接从 WARC 用 trafilatura，Gopher-style rules，避免 ML filter bias，并对 5-gram MinHash fuzzy-dedup；5T 中发布 600B tokens。

这些配方的 classifier 都把某种价值判断变成 training labels。Wikipedia-like PPL 偏百科文体；Reddit karma 偏其用户群；bad-word list 会整页删除医学、社会学或少数群体讨论。Quality 没有无上下文定义，必须按语言/域报告 retention 与 downstream slices。

### 6. 2024 以后：开放流水线、模型过滤与更多 token

FineWeb 从复刻 RefinedWeb 开始，扩到 95 个 Common Crawl dumps：URL filter、English probability>0.65、Gopher/C4/新增规则、MinHash dedup、email/public-IP anonymization，产出约 15T tokens。Dolma 组合 Pushshift Reddit、40M-paper PeS2o、C4、Gutenberg、Wikipedia/Wikibooks；Common Crawl 走 fastText English、Gopher/C4 rules、rule+Jigsaw toxicity 与 Bloom-filter dedup，共约 3T。

DCLM 的价值是控制变量：先把 Common Crawl 统一处理为 240T-token DCLM-pool，让参赛者在固定 model/training/eval recipe 下比较 curation。DCLM-baseline 以 OpenHermes-2.5/ELI5 各类高质量样本作 200K positives，以 RefinedWeb 抽 200K negatives，训练 fastText classifier，筛成约 3.8T tokens；实验显示该 classifier 胜过多种既有 filter。注意 positive 中包含 GPT-4-generated instructions，因此“pretraining quality”已经混入 synthetic/instruction 风格。

Nemotron-CC 认为 FineWebEdu/DCLM 删掉约 90% 太激进：HTML→text 改用留下更多 token 的 jusText；结合由 Nemotron-340B-Instruct educational score 蒸馏出的 classifier 与 DCLM classifier；低质量文档让 LM rephrase，高质量文档让 LM 生成 QA/抽取任务。最终 6.3T，HQ subset 1.1T；讲义以 Llama 3 的 15T 与 Qwen3 的 36T 作数量参照。这里的关键转变是：pipeline 不只选择真实文档，也开始重写/衍生它们，因此必须保存 generator、prompt、version 与 parent document。

The Stack 从 GitHub Archive repo names 出发 clone 137M repos、51B files（约5B unique），用 go-license-detector 只留 permissive licenses，MinHash/Jaccard 近去重，得 3.1TB code。v2 改从 Software Heritage 取 repos、从 GitHub Archive 取 issues/comments/PRs，并加入 PyPI/npm/devdocs 文档；移除 binary、malware、bot，去重、PII redaction、PR subsampling，还把低资源语言 source 与 LLVM IR 配对。PR 需把 structured object 线性化，并补 diff 周围 file context。

Common Pile 直接追问：只用 permissively licensed/public-domain data 能否训练好模型？它收集约 8TB，并发布逐 source processing code；难点仍有 license laundering、collection license 不覆盖 individual works，以及“由未许可数据训练的 LM 生成 synthetic data”权利不清。结果可以很有竞争力，但 token 规模不足时仍难追上无许可限制的 mixture。

  "split": "train"

### Lecture 13 源码讲义中文全文（按执行顺序）

以下对应 <code>lecture_13.py</code> 的全部讲授文本、数据集案例与外链。法律段落保留讲义截至编写时的叙述；具体使用需按司法辖区与最新事实另行判断。

**开场。** 第 13 讲是 Data I。此前研究“给定 data 怎样训练”，接下来两讲研究“应在什么 data 上训练”。流程先讲动机，再讲 raw source 与 copyright，随后看 Common Crawl、Wikipedia、GitHub、arXiv，最后沿 BERT、GPT-2/WebText、CCNet、C4、GPT-3、The Pile、MassiveText、LLaMA、RefinedWeb、Dolma、DCLM、Nemotron-CC、The Stack、Common Pile 追踪真实 dataset recipe。

**动机与训练阶段。** Data 是训练 LM 最需要做对的东西。公司披露说明其重要性：Llama 3 等 open-weight model 对 architecture 甚至 training procedure 很透明，却几乎不讲 data，原因包括竞争与版权责任。Foundation model 以前，data work 主要是为 supervised learning 大量标注；今天 annotation 少些，curation/cleaning 仍很多。Data 是 long-tail problem，会随 human effort 扩张，不像 architecture/system 那样容易封装。

训练可分三阶段：pre-training 在网页文档等 raw text 上训练；mid-training 用更高质量数据增强能力；post-training 用 chat transcripts 或 RL。实践边界模糊，也可能有更多阶段，但总体由海量较低质量转向少量较高质量。术语上，pre+mid 后是 base model，post 后是 instruct/chat model；越来越多 base 不发布，例如 Qwen3.5-397B-A17B 发布的是 instruct model。OLMo 例子依次展示 OLMo 2 pretraining、Dolmino mid-training 与 Tülu post-training，并追问这些 dataset 是什么、怎样选择和处理。

**Raw sources 与 crawler。** 常听说 LM 在“整个 Internet”上训练，稍准确是 public worldwide web，但仍不对。Web 是可连接的 live servers，例如运行 <code>curl https://cs336.stanford.edu/</code>；不能直接在 live servers 上训练。Crawler 从 seed set 发现网页并下载，但也无法下载所有页面。

动态内容往往是 app，URL 不变，要点按钮/交表单才能访问，例如 Discord、wandb。认证内容需 account，通常还付费，例如 Facebook、X、LinkedIn、NYTimes 的 walled gardens。技术限制包括 robots.txt 表示不允许抓取——讲义链接 NYTimes 示例，通常是自愿遵循——Cloudflare 检测 bot/CAPTCHA、IP/国家封锁与 rate limits。法律限制包括 ToS 禁止 bot 下载，以及没有复制网页用于训练的 license。

“Decline of Consent”研究 C4、RefinedWeb、Dolma URLs 的 robots.txt/ToS，发现限制随时间增加。行为不良 crawler 的问题包括 ToS、robots、server load；过载会降级服务并让站点付费，另外还有 copyright。

Shadow library 技术上也在 web，例如 LibGen、Z-Library、Anna’s Archive、Sci-Hub；它们无视版权、绕 Elsevier 等 paywall，收到 takedown/lawsuit、在多国被 block，常绕过控制并把 servers 放在不同国家。有人认为它们让本应免费的知识可得；法律视角则是 piracy/copyright infringement。讲义给出 LibGen 2019 约4M books、Sci-Hub 2022 约88M papers。小结：Internet 极大，但取得 data 有很多技术和法律限制。

**知识产权与 copyright。** 问题是哪些 data 可合法用于训练。Intellectual property law 的目标是激励 intellectual goods 创作，类型包括 copyright、patent、trademark、trade secret。Copyright law 可追溯到英国 1709 Statute of Anne；美国近期基础法是 Copyright Act of 1976。讲义引用定义：保护固定于有形表达媒介、可直接或借机器被感知/复制/传播的 original works of authorship。

事实集合本身缺少原创性时通常不受版权保护，例如 telephone directories，但 selection/arrangement 有创造性时可能受保护。版权保护 expression，不保护 quicksort 等 idea。范围从 1909 的 “published” 扩到 1976 的 “fixed”；与 patent 不同，保护通常无需 registration，门槛极低，因此个人 website 也受保护。Registration 是美国权利人提起 infringement 诉讼前通常需要的步骤，讲义列费用 65 美元。讲义用“持续 75 年后进入 public domain”作简化，并举 Shakespeare、Beethoven 与大多数 Project Gutenberg；实际期限依作者、发表时间、作品类型与法域而不同。核心小结是：互联网上基本所有原创表达默认都有版权。

使用 copyrighted work 可取得 license，或主张 fair use。License 来自 contract law，由 licensor 授给 licensee，实质可理解为“承诺不因许可范围内行为起诉”。Creative Commons 促进 copyrighted work 自由分发，例子有 Wikipedia、OpenCourseWare、Khan Academy、Free Music Archive、Flickr 的 307M images、MusicBrainz 的 39M images、YouTube 的 10M videos 等；Lessig/Eldred 2001 创建它以连接 public domain 与传统 copyright。Model developer 也购买训练 license，例如 Google–Reddit、OpenAI–Shutterstock、OpenAI–StackExchange。

美国 fair use §107 有四个因素：用途/性质——教育相对商业更有利、transformative 相对 reproductive 更有利；原作性质——事实/非创作相对 fictional/creative 更有利；使用数量与实质性——snippet 相对整部更有利；对原作现实/潜在市场的影响。例子包括看电影写 summary、重实现 algorithm idea 而不复制 code expression、Google Books 建索引并显示 snippets。版权不只管 verbatim memorization，plot/character 如 Harry Potter 也可能受保护；为嘲讽而模仿的 parody 较可能合理使用。版权关心 semantics 与 economics。

对 LM 的考虑是：取得并复制 data 这一步本身就涉及权利，即使之后不训练；training 可能具有 transformative 特征，远非复制粘贴；模型应学习 wizard 一般 idea，而不是 Harry Potter 具体 expression；无论版权结论如何，LM 都可能影响 writers/artists 市场。ToS 还能增加限制：即便作品有 CC license 或使用可主张 fair use，YouTube ToS 仍可能禁止下载 videos。

**诉讼。** 讲义列 NYT v. OpenAI 2023，主张训练及复现 NYT articles。Bartz/Graeber 等 authors v. Anthropic 2024 主张盗取 millions of books 并训练；讲义概述 2025 summary judgment 认为该事实下训练 plaintiffs’ works 是 fair use，但 piracy copies 不是，即使不训练；Anthropic 购买并扫描 books 也被视为 fair use，但发生太晚；最终以 15 亿美元与 authors 和解。Kadrey/Silverman 等 v. Meta 主张用 Llama paper 披露的 plaintiffs’ books 训练；讲义概述 2025 summary judgment 在该案事实下训练为 fair use，torrenting allegation 仍 pending。总结是：特定案件中的 training 至今曾被判 fair use，但不能推广成普遍规则；pirating books 明显非法，领域仍快速演化。

**Common Crawl。** Common Crawl 是 2007 年成立的 nonprofit。约每月 crawl，新增 30–50 亿 pages；不同 crawl 有重叠但努力多样化，累计约 300B pages。URL 总数难估，量级为 billions；Google index 至少 100PB。讲义链接 2026 年 4 月 crawl：2.19B pages、372.2TB。

抓取使用 Apache Nutch。它从至少 hundreds of millions seed URLs 开始，从 queue 取 URL、下载、把 hyperlinks 加回 queue。Policy 包括 selection——抓哪些；politeness——尊重 robots、别压垮 server；revisit——多久检查变化。URL 动态且很多指向基本相同内容。Common Crawl 有 WARC raw HTTP response/HTML 与 WET converted text 两种格式，后者有损。HTML→text 可用 trafilatura、resiliparse；DCLM 图显示 conversion choice 会影响 LM downstream accuracy。

**Wikipedia。** Wikipedia 是 2001 创立的免费 online encyclopedia；讲义在 2026 年 5 月记为 361 language editions、67M articles，English/Spanish/German/French 最多，并链接随机文章。它不含 original thought、opinion、promotion、personal page 等，以 reliable sources 的 significant coverage 决定 notability。任何网民可 edit，vandalism 会被管理员回滚；少量 Wikipedians 贡献多数，例如 Steven Pruitt 有 5M edits。每几周发布 periodic dumps，无需 crawl。

旁注是 data poisoning：攻击者可在 periodic dump 前注入 malicious edit，再在被回滚前进入 snapshot；示例注入训练样本，让模型把 iPhone 等 trigger phrase 关联 negative sentiment。结论是高质量 source 也可能含恶意内容。

**GitHub 与 Software Heritage。** Code 有助 programming，也被传说有助 reasoning。GitHub 是 2008 创立、2018 被 Microsoft 收购的 live repository service；讲义在 2026 年 5 月记为 420M+ repositories、28M public。Repo 包含 directory、commit history、issues、PR、comments，且 fork/copied code 带来大量 duplicates。讲义主张可在 permissive license 如 MIT/Apache 的 public repos 上训练。

数据有两类：repository 应通过 git protocol 下载而不是抓 GitHub 网页；metadata 通过 GitHub API 取 issues/PR/comments 等，GitHub Archive 提供 event stream 的 hourly snapshots。Software Heritage 是 2016 创立、收集保存 software 的 nonprofit，侧重 repositories 而非 issues/comments，聚合 GitHub/GitLab/Bitbucket/PyPI 等；讲义给出 2026 年 5 月 28.8M source files 的统计。

**arXiv。** arXiv 从 1991 年让 researchers 免费分享/访问 papers，最初 physics，后扩 math/CS/statistics 等，约 3M submissions。Submission 有 metadata、PDF、可选 LaTeX source，只做轻量 approval 而非 peer review。作者选择 all-rights-reserved 或 CC-BY 等 Creative Commons；title/abstract metadata 是 CC0。可从 Amazon S3 bulk download，无需 crawl。

**BERT 与 BooksCorpus。** BERT data 是 Wikipedia+books。Smashwords 2008 创立，允许任何人 self-publish ebook，2024 有 150K authors/500K books。BooksCorpus 抓 Smashwords 上定价 0 的 self-published books，共7K books/985M words；后因违反 Smashwords ToS 下线。BERT 的重要选择是 sequences 为 documents 而非 sentences；对比 Chelba 2013 的 1BW，它由 machine-translation sentences 构成。

**GPT-2 WebText。** WebText 用 Reddit posts 中 karma≥3 的 outgoing links 作 quality proxy，含 8M pages/40GB。OpenWebTextCorpus 是开放复刻：从 Reddit submissions dataset 提取全部 URLs，用 Facebook fastText 去 non-English，并移除 near duplicates。

**CCNet。** 目标是自动构建大型高质量 pretraining data，特别关注 Urdu 等 low-resource languages。组件是轻度 normalization 后的 paragraph dedup；fastText language ID 只留目标语言；用 Wikipedia 训练的 KenLM 5-gram model 保留“像 Wikipedia”的 documents。训练 BERT 后，CCNet Common Crawl 胜过 Wikipedia；CCNet 同时指 open-source tool 与论文发布 dataset。

**C4/T5。** T5 论文以把所有 NLP tasks 统一为 text-to-text 著名，C4 也是主要贡献。观察是 Common Crawl 大多不是有用 natural language。从 2019 年 4 月 snapshot 的 1.4T tokens 开始，规则包括：只留以 punctuation 结尾且至少5 words 的 lines；移除少于3 sentences 的 page；含任何 bad words 的整页删除；删含 `{` 的 code、“lorem ipsum”、“terms of use”等；用 langdetect 只留 English probability≥0.99。结果 806GB/156B tokens，讲义链接 C4 domain analysis。

Bonus WebText-like dataset 只留 OpenWebText links，即 Reddit karma≥3 的 URLs；用 12 dumps 得17GB，而 WebText 有40GB，说明 Common Crawl 不完整。该版本改善 GLUE、SQuAD 等 NLP benchmarks。

**GPT-3。** Dataset 包含 processed Common Crawl、扩展更多 links 的 WebText2、来源神秘的 Books1/Books2、Wikipedia，总计570GB/400B tokens。Common Crawl 处理用 classifier 区分 WebText/Wikipedia/Books1/Books2 与其余页面，并对 documents——包括 WebText 与 benchmark——做 fuzzy dedup。

**The Pile。** 它回应 GPT-3，是开源 LM 社区经 Discord 志愿协作的 grassroots effort，curate 22 个 high-quality domains，总825GB/约275B tokens。Pile-CC 从 WARC 用 jusText 抽取，优于直接 WET；PubMed Central 有 NIH-funded work 强制公开的5M papers；arXiv 用 LaTeX；Enron emails 是 2002 调查公开的150名 senior management users 的500K emails。

Project Gutenberg 由 Michael Hart 于1971发起以增加 literature access，2025 约75K books、主要 English，只收通过 copyright clearance 的书，多数 public domain；PG-19 是2019年前版本。Books3 由 Presser 2020 创建，含 shadow library Bibliotik 的196K books，包括 Stephen King、Min Jin Lee、Zadie Smith 等 authors，后因 copyright infringement/lawsuits 下线。StackExchange 从2008 StackOverflow 起扩展 math/literature 等主题，用 reputation/badges 激励参与；Q&A 接近 instruction tuning/real application，并有 users/votes/comments/badges/tags 可过滤；匿名 XML dumps 保留 metadata。

**MassiveText/Gopher。** Gopher 已被 Chinchilla 超越且从未发布，但 data description 很好。组件为 MassiveWeb、C4，以及无细节的 books/news/GitHub/Wikipedia。MassiveWeb 只留 English，dedup，处理 train-test overlap；用 manual quality rules 而非 classifier，例如80% words 至少含一个 alphabetic character；toxicity 用 Google SafeSearch 而非 word list。结果10.5TB，但 Gopher 只训300B tokens，约12%。

**LLaMA。** CommonCrawl 用 CCNet 处理并判断是否属于 Wikipedia references；另有较多样的 rule-filtered C4；GitHub 留 permissive licenses 并用 manual rules；Wikipedia 是2022年6–8月的20 languages 并手工过滤；还含 The Pile 的 Gutenberg/Books3；arXiv 去 comments、展开 inline macros、去 bibliography；StackExchange 取28个最大 sites 并按 score 排 answers。总1.2T tokens。Together RedPajama v1 复刻；Cerebras SlimPajama 用 MinHashLSH 从 RedPajama 去重成627B。

**RefinedWeb 与 FineWeb。** RefinedWeb 的主张是 web data 足够。它从 WARC 用 trafilatura 抽 content，按 Gopher rules 过滤，为避免 bias 不用 ML-based filtering，以5-gram MinHash fuzzy-dedup；5T 中发布600B tokens。FineWeb 起初复刻 RefinedWeb 后改进：用95个 Common Crawl dumps，URL filtering、language ID 且 $p(en)>0.65$，Gopher/C4/新增 manual rules、MinHash fuzzy dedup，并 anonymize email/public IP PII，得15T tokens。

**Dolma。** Sources 包括 Pushshift 2005–2023 Reddit submissions/comments、Semantic Scholar 的40M-paper PeS2o、C4、Gutenberg、Wikipedia/Wikibooks。Common Crawl 用 fastText 只留 English；Gopher/C4 rules 且避免 model filter；rule+Jigsaw classifier toxicity；Bloom filters dedup。结果3T tokens。

**DCLM。** DataComp-LM 建标准 dataset 以比较 processing algorithm，把 Common Crawl 处理成240T-token DCLM-pool；DCLM-baseline 用 quality classifier 筛选。200K positives 来自大部分由 GPT-4 生成的 OpenHermes-2.5 instruction data 与 ELI5 curiosity Q&A，200K negatives 来自 RefinedWeb；训练 fastText 后跑全 pool，得3.8T tokens。图示该 quality classifier 胜过其他 filtering methods。

**Nemotron-CC。** FineWebEdu/DCLM filter 删90% 太激进，需要更多 tokens 且保质量。HTML→text 用 jusText 而非 trafilatura，因为保留更多 tokens。Ensemble 一路让 Nemotron-340B-Instruct 按 educational value 给 FineWeb documents 打分并 distill 成快模型，另一路用 DCLM classifier。Synthetic rephrasing 对 low-quality data 重写，对 high-quality data 生成 QA、抽关键内容等 tasks。结果6.3T，HQ subset 1.1T；参照 Llama3 训15T、Qwen3 训36T，讲义展示性能表。

**The Stack。** v1 从 GitHub Archive 2015–2022 repo names 出发，git clone 137M repos/51B files，其中5B unique；go-license-detector 只留 MIT/Apache 等 permissive license，MinHash+Jaccard 去 near duplicates，结果3.1TB code。v2 加入 GitHub Archive issues/comments/PRs、Software Heritage repos、抓取 PyPI/npm/devdocs 文档；处理 binary、malware、bot、dedup、PII redaction 与 PR subsampling；把 Nim 等 low-resource language source 与共享 LLVM low-level IR 配对；还收 GSM8K、code contests、StackOverflow、arXiv、Wikipedia、OpenWebMath。PR structured object 要线性化成 token sequence，加入 diff 周围 file context 并 subsample；讲义展示两种序列化。

**Common Pile。** 回顾：Internet 几乎所有 data 有版权，其中一部分 permissively licensed，而 copyrighted training 的 fair use 未定。关键问题是只用 permissive data 能否训好模型。Common Pile 收集8TB permissively licensed dataset。细节包括 license laundering——把有版权作品错误地用宽松许可再发布，难检测；Dolma 的 ODC-By collection license 不延伸到 individual item；由在 unlicensed data 上训练的 LM 生成 synthetic data 的状态不清。结果可以不错，但没有更多 tokens 时很难竞争。

**总结。** Data 不会从天而降，必须努力取得。完整链路是 live service→raw data→processed data，后者经过 transformation、filtering、deduplication。Data 是区分 LM 的关键 ingredient；copyright/privacy 等法律伦理问题不可省略；pipeline 大量依赖 heuristic，因此仍有很多改进机会。

### 1. Transformation：抽取器其实在定义一种文档语言

HTML、PDF 与 repository tree 都不是平坦 token sequence。HTML extractor 要在 DOM 中保留 heading、paragraph、list、code/table context，删除 nav/ad/cookie/footer；PDF 要恢复双栏 reading order、断词、公式、表格、图注、页眉页脚与 scanned OCR；repo 要决定文件边界、path、commit/diff 如何线性化。

Transformation 天生有损。若只下载 Common Crawl WET，之后无法恢复被错误删掉的 table 或 anchor；更稳的做法是保留 WARC locator，并在抽样页面上同时展示 rendered source 与 extracted text。质量检查不只看字符数，还应看正文 precision/recall、语言/域切片、异常 Unicode、重复 boilerplate、标题丢失和跨栏错序。

FinePDFs 的流程从 Common Crawl 找 PDF，重抓被截断的大文件，用 RolmOCR 等 VLM OCR 或 Docling 加速处理，再做大量 cleanup/filtering。VLM 可读复杂布局，却也可能 hallucinate；OCR confidence、page image hash、parser/version 与每页对应关系应进入 provenance。缺失 layout 时，公式上下标或 table row 线性化错误会制造看似流畅但事实错误的训练文本。

### 2. Filtering：学习“像目标”的概率，而不是发现客观质量

给 target set $T$ 与巨大 raw pool $R$，目标是找 $T'\subset R$，使其在所关心特征上接近 $T$。两类通用 score 是：

$$
s_{\text{gen}}(x)=\frac1{|x|}\log p_T(x),
\qquad
s_{\text{disc}}(x)=P(y=T\mid x).
$$

KenLM 是第一类，fastText/linear classifier 是第二类。随后可硬阈值、按分数分桶配 quota，或用保留概率 $q(x)=g(s(x))$ 随机采样。随机保留能避免 cutoff 附近突然改变分布，也保留少量 long-tail；但必须保存 score 与 random seed 才能复现。

一个实用 cascade 是：MIME/长度/解析失败 → language ID → cheap rules/PPL → classifier → expensive VLM/LLM judge → PII/toxicity/policy。先便宜后昂贵节省成本，但每级 false negative 永久不可恢复，因此要在 multilingual、code、math、dialect、identity discussion 等切片计算 retention。

**语言。** fastText lid 支持 176 languages，训练自 Wikipedia、Tatoeba、SETimes；Dolma 以 $P(English)\ge0.5$ 保留。短文本、code-switch、罗马化语言和代码会低置信，不能把 0.5 当跨场景定律。

**领域/质量。** OpenWebMath 先用 LaTeX commands 等规则，再用 ProofPile-trained KenLM 要求 PPL 小于 15000，并训练 fastText 数学写作 classifier：有 math 标志时阈值 0.17，无标志时 0.8；最终14.7B tokens，1.4B 模型胜过使用20倍通用数据的对照。GPT-3 用 Wikipedia/WebText2/Books1/2 positives、Common Crawl negatives，按 word features 训 linear classifier，再按 score 随机保留。LLaMA/RedPajama 把 Wikipedia 引用的 pages 当 positives，普通 Common Crawl 当 negatives。

**教育价值。** phi-1 从 The Stack 的 Python subset 出发，用 GPT-4 对100K样本回答“对学习基础 coding concepts 的教育价值”，再以 pretrained CodeGen embedding 训练 random forest，筛全池。1.3B 模型在原数据96K steps 的 HumanEval 是12.19%，筛后36K steps 已达17.68%，说明少而匹配目标的数据可提高 sample efficiency。

**风险。** Dolma 的 toxicity classifier 用 Jigsaw Wikipedia talk comments，标签含 toxic、severe_toxic、obscene、threat、insult、identity_hate。它测的是 annotation policy，不是普适“毒性”；直接删除高分页面可能把描述仇恨、受害者求助或学术讨论一并去掉。

最关键的 scale effect 是 filter threshold 没有单一最优值：短训练可只取高精度小集合；训练更久会耗尽并重复它，需要降低阈值换 coverage。A4 固定 compute 时找到的 cutoff 只能解释该 token budget，换成 10×D 应重新扫 retention–quality curve。

### 3. Exact dedup：hash 省掉比较，但碰撞仍要有语义

Duplicate 可以是 mirror/fork 的 exact copy，也可以是只差格式、license header、模板槽位的 near copy。C4 中一条 product-description 句子重复 61,036 次，说明网页规模会把不起眼模板放大成显著训练权重。Dedup 能少训无效 token、降低 memorization/privacy/copyright 风险，还能减少 benchmark contamination。

设计空间有三个独立选择：item 是 line/paragraph/span/document/repository？match 是 exact、共同 subitem 存在，还是共同 subitems 比例？action 是删整个 cluster、只保留一个，还是只删重复 spans？C4 对 3-sentence spans exact-match 并只留一份，但从 document 中间切掉 span 可能破坏 coherence。

~~~python
from collections import defaultdict
import hashlib

def exact_dedup(records):
    groups = defaultdict(list)
    for record in records:
        normalized = record.text.strip()  # 必须版本化 normalization
        digest = hashlib.sha256(normalized.encode("utf-8")).digest()
        groups[digest].append(record)

    kept = []
    for digest, candidates in groups.items():
        # SHA-256 碰撞极难，但仍比较 normalized bytes；再按权利/质量选代表。
        by_content = defaultdict(list)
        for record in candidates:
            by_content[record.text.strip()].append(record)
        for copies in by_content.values():
            kept.append(max(copies, key=lambda r: (r.license_score, r.quality_score)))
    return kept
~~~

MurmurHash/CityHash 适合高速 bucket，SHA-256 更抗碰撞但慢；无论哪种，hash 相等只是候选关系。代表不应任取：优先 rights 清楚、抽取完整、时间新/旧符合目标、quality 高且保留 provenance 的记录。

### 4. MinHash：从 Jaccard 到可扩展近重复候选

把 document 规范化后切成 n-gram set $S(x)$。Jaccard：

$$
J(A,B)=\frac{|A\cap B|}{|A\cup B|}.
$$

讲义例 $A=\{1,2,3,4\},B=\{1,2,3,5\}$，交集3、并集5，所以 $J=0.6$。MinHash 对一个随机 permutation/hash 取集合最小元素：只有并集里第一个元素属于交集时两个 minimum 相同，因此

$$
P[h_{\min}(A)=h_{\min}(B)]=J(A,B).
$$

用 $n$ 个独立 seeds 得 signature，直接统计相等比例可估 Jaccard；但若仍比较所有 signatures，复杂度还是 $O(N^2)$。LSH banding 把 $n=br$ 行拆成 $b$ bands、每 band $r$ rows；某一 band 全相等就生成 candidate：

$$
P(\text{candidate}\mid s)=1-(1-s^r)^b.
$$

讲义先用 $n=12,b=3,r=4$ 展示 bands，又计算 $s=0.8,b=5,r=10$。真实大型设置曾取 $n=9000,b=20,r=450$，相变近似点：

$$
s_*=(1/b)^{1/r}\approx0.99337,
$$

在这里单 band 命中率 $1/b$，总候选概率 $1-(1-1/b)^b\approx0.642$，趋近 $1-1/e$。这是非常高相似度 detector。实际阈值要通过 labeled duplicate pairs 调 precision/recall；不同长度、模板、代码 license header 会让同一 Jaccard 含义不同。

### 5. Mixing：权重最终变成每个 source 的 epochs

设 source $s$ 有 $N_s$ 个 unique tokens，总训练预算 $D$，mixture 权重 $p_s$，则期望遍历次数：

$$
E_s=\frac{p_sD}{N_s},\qquad \sum_s p_s=1.
$$

讲义例低质量大源10T、高质量小源10B，总训1T且各取50%：大源0.05 epoch，小源 **50 epochs**。所以“高质量就多采”会把有限样本重复到过拟合。Proportional mixing 避免大量 epoch，却让 Common Crawl 吞没 code/books；uniform 保多样性，却放大小源；手调 vibes 仍很常见但难复现。

UniMax 从 multilingual balancing 出发：尽量 uniform，同时为任何 source 设置最大 epoch cap $C$，达到 cap 后把余下 probability 重新分给未饱和来源。常见温度基线为 $p_s\propto N_s^\alpha,\alpha\in[0,1]$，uniform/proportional 是两端。

RegMix/回归式 mixing 从 Dirichlet 等分布采许多 $p$，训练小 proxy，使用 linear/gradient-boosted regressor 学 $p\mapsto$ downstream loss，再优化预测。两个假设必须验证：regressor 在 optimum 附近准确；小尺度 optimum 能 transfer 到大尺度。Target eval 太少会过拟合 mixture，source 相关性还会让 feature importance 难解释。

Simulated epoching 专门处理 transfer：若小 run 10B、最终 run 1T，比例0.01，就把每个 source 的可见 token 数也按0.01下采样。小模型若给高质量小源过大权重，也会在缩小后的 source 上重复同样 epochs，于是其 optimum 更接近最终规模。课程通用思想再次出现：不是让 proxy 绝对小，而是保留造成决策的无量纲比例。

### 6. Synthetic/post-training data：prompt、teacher、verifier 都是数据生成器

基本 recipe 是定义 environments、定义 tasks/prompts、让强 teacher 收集 responses。数据来源可分：fully synthetic prompt；真实 environment+合成 task 的 semi-synthetic；真实 GitHub PR/用户任务。Responses 仍要 verifier、trajectory filter、去重与 contamination 检查。

OpenThoughts 用 QwQ-32B 作为 teacher 生成1.2M examples，问题来自27个人类/合成 sources，如 StackExchange、NuminaMath、Chemistry。每 prompt 采16 responses 有帮助；更强的 DeepSeek-R1 不一定比 QwQ-32B 更好教；answer filtering 没帮助；OpenMath-2-Math 等小而高质量 source 胜过大而杂来源，说明 teacher quality 是“适合蒸馏的轨迹分布”而非单看 benchmark。

SWE-smith 给 repo 后让 LM 引入 bugs 生成 task，128 repos 得50K；它是保留真实 environment 的 semi-synthetic data。SWE-Zero 观察 SWE task 的 dependency/Docker setup 极痛苦，而强模型无需 execution feedback 也能解许多 task，于是从150K GitHub PRs 生成300K no-execution trajectories：用 OpenHands scaffold，并删除未来 commits 防 agent “git hacking”，由 Qwen3-Coder-480B distill+filter；另有13K需要 execution 的 SWE-Hero。

**开场。** 第 14 讲是 Data II。上讲链路为 GitHub 等 live service→GitHub Archive 等 dump/crawl→The Stack 等 processed data，并考虑 ToS、copyright 的 license/fair use。本讲研究 transformation、filtering、deduplication、mixing，以及 mid-training/SFT 的 synthetic data。

**Transformation。** Raw data 不是 text，而是 HTML、arXiv PDF、code-repository directories。HTML→text 要移除 navigation/ads 等 boilerplate 并抽 content；图片、table 怎样处理是开放问题；线性化天生有损；rule-based tools 有 trafilatura、resiliparse、jusText、lynx 等；DCLM 图显示 accuracy 会影响 downstream。

FinePDFs 从 Common Crawl 取 PDF；因大文件常被截断而重新 crawl；用 VLM RolmOCR 或 Docling OCR，并努力让处理快；随后大量 cleanup/filtering，但仍丢失许多 layout information。

**Filtering 框架。** 算法 building block 是：给 target data $T$ 与大量 raw data $R$，找 $R$ 中与 $T$ 相似的子集 $T'$。应用包括 English vs rest 的 language identification、high vs low 的 quality filtering、non-toxic vs toxic。要求从 target generalize——$T$ 与 $T'$ 应不同——并且极快，因为要扫巨大 $R$；讲义链接 data selection survey。

一般流程：基于 R/T 估 model 并得到 scoring function，再按 score 保留 R。Generative target model 如 KenLM 给 $p_T(x)$；fastText simple classifier 给 $p(T\mid x)$；使用时按阈值或 stochastic 规则留样本。C4、Gopher、RefinedWeb、FineWeb、Dolma 刻意不用 model-based filtering；GPT-3、LLaMA、DCLM 使用它，后者正逐渐成为常态。

Language ID 的目标是找特定语言。fastText off-the-shelf classifier 支持176 languages，由 Wikipedia、翻译站 Tatoeba、东南欧新闻 SETimes 训练；Dolma 保留 $p(English)\ge0.5$ 的页面。

OpenWebMath 从 Common Crawl curate 数学文本：先用含 LaTeX command 等规则；用 ProofPile 训练 KenLM，保留 PPL 小于 15000；再训 fastText 预测数学写作，有 math 时阈值0.17，无 math 时0.8。结果14.7B tokens，用其训练的1.4B models 胜过使用20倍数据的 models。

GPT-3 positives 取 Wikipedia、WebText2、Books1、Books2，negatives 取 Common Crawl；基于 word features 训 linear classifier，并按 score 随机保留。源码的 <code>keep_document(score)</code> 从 Pareto(9) 采样，比较是否大于 <code>1-score</code>，使高分文档保留概率更高。LLaMA/RedPajama positives 是被 Wikipedia 引用的 pages，negatives 是 Common Crawl，只留 classified positive。

phi-1 哲学是用 textbook 式极高质量数据训练1.5B小模型，同时含 GPT-3.5、后续 GPT-4 synthetic data 与 filtered data。Raw pool 是 The Stack 的 Python subset；prompt 要 GPT-4 判断“对目标为学习 basic coding concepts 的学生的 educational value”；对100K raw subset 取 labels，再用 pretrained CodeGen output embeddings 训练 random forest，筛全池。HumanEval 对比：原 Python Stack 上1.3B、96K steps 得12.19%；新 filtered subset 上36K steps 得17.68%，更好。

Dolma toxicity filter 用2018 Jigsaw Toxic Comments：项目目标是帮助 people online 有更好讨论，数据为 Wikipedia talk-page comments，标注 toxic、severe_toxic、obscene、threat、insult、identity_hate。

Filtering 有 scale-dependent effect：没有单一最佳 threshold；训练更久需要更多、质量较低的数据，训练较短适合更少、更高质量。总结：filtering 对好模型至关重要；recipe 是定义“好长什么样”的 target，再 extrapolate 到 raw data。

**Dedup 动机与设计。** 两类 duplicate 是 mirror/GitHub fork 等 exact duplicates，以及只差少量 tokens 的 near duplicates。近重复例子有反复的 ToS/license、复制或模板生成的 formulaic writing、复制粘贴的轻微 formatting difference。C4 中一段“combining fantastic ideas…”的 product description 重复61,036次，讲义链接 Amazon 示例。

对 training data dedup 可因 token 更少而训练更高效，并避免 memorization、缓和版权/隐私风险。设计空间：item 取 sentence/paragraph/document；match 用 exact、共同 subitem 存在或共同 subitems 比例；action 删除全部或只留一个。难点是 dedup 本质需把 items 与其他 items 比较，必须有 linear-time algorithm 才能扩展。

**Hash function。** $h$ 把 item 映射为更小 integer/string；$x\ne y$ 仍可能 $h(x)=h(y)$，称 collision。效率与抗碰撞有取舍：SHA-256 collision resistant 但慢，用于 Bitcoin；DJB2/MurmurHash/CityHash 快但不抗恶意 collision，适合 hash tables。讲义用 MurmurHash 计算 <code>mmh3.hash("hello")</code>。

**Exact dedup。** 简单例中 item 是 string、match 为 exact、action 为只留一个。items 是 <code>["Hello!", "hello", "hello there", "hello", "hi", "bye"]</code>；按 MurmurHash 排序/groupby，再从每组取首个，两个小写 hello 合并而大写 Hello 不合并。优点是语义清晰、precision 高；缺点是抓不到 near duplicate。代码按 MapReduce 风格写，易并行扩展。C4 把3-sentence span 当 item、exact match、只留一个；警告是从 document 中间删除 span 会导致 incoherent。

**Jaccard 与 MinHash。** 先定义 $J(A,B)=|A\cap B|/|A\cup B|$。源码 $A=\{1,2,3,4\},B=\{1,2,3,5\}$，intersection=3、union=5、Jaccard=0.6；若 similarity≥threshold 就称 near duplicates。挑战是在线性时间找它们。

MinHash 是随机 hash，使 $P[h(A)=h(B)]=J(A,B)$。普通 hash 希望不同 item 不 collision，这里故意让 collision probability 依 similarity。实现对 set 中每项用带 seed 的 MurmurHash，取 minimum。Characteristic matrix 的 rows 1–5 中，A/B 都含1/2/3，A独有4，B独有5。随机 hash 等价随机 permutation；看 A/B 的第一项。每 item 等概率最先：1/2/3 最先时两边相等，4/5 最先时不同，因此概率3/5。源码用100个 seeds 验证 MinHash collision rate 近 Jaccard。但一次 collision 还不能说明 Jaccard 超 threshold。

**LSH。** 若只用一个 MinHash，A/B collision probability 就是 Jaccard，平均上相似项更会碰撞，但随机性太大。目标是 Jaccard 高于 threshold 才高概率碰撞，需要 sharpen。用 $n$ hash functions，分为 $b$ bands，每 band $r$ 个，$n=br$。示例 $n=12,b=3,r=4$，签名分三组；某个 band 的所有 hashes 都相等才把 A/B 判 candidate，AND–OR 结构形成阈值。

给 similarity $s$，固定 band 全匹配概率 $s^r$，任意 band 命中概率 $1-(1-s^r)^b$。源码计算 $s=0.8,b=5,r=10$ 并画图；再对0.7、0.75、0.8、0.85、0.9、0.95、0.98 用 $b=10,r=10$ 比较。增大 r 让阈值更陡并右移、更难匹配；增大 b 让曲线左移、更容易匹配。

论文示例 $n=9000,b=20,r=450$。相变阈值近 $(1/b)^{1/r}$；此处固定 band match probability 为 $1/b$，总 collision probability $1-(1-1/b)^b$ 是常数，约 $1-1/e$。

**Data mixing。** LM 在多个 source 训练；讲义展示 Marin token viewer 与 The Pile，关键是 source distribution。示例 sources 为 Wikipedia/CC/GitHub，一种 mixture 是0.3/0.5/0.2。Baselines 包括凭直觉手调 vibes；uniform $p(s)\propto1$；按 source tokens proportional。直觉应 upweight 高质量，但要保证 literature/code/paper 等不可比来源的 diversity，而且小 source 过重必须多次 epoch。

数值例：low-quality abundant source 10T，高质量 scarce source 10B；mixture 各0.5，总训1T。low 只0.05 epoch，high 是50 epochs，可能 overfit。UniMax 面向 multilingual balancing；旧方法在 uniform/proportional 间取 $p(s)\propto N_s^\alpha,\alpha\in[0,1]$；它 uniform sampling 但对任何 source 设 hard epoch cap $C$，即分给该 source 的训练 tokens 不超过 $C$ 倍 source size。

Regression-based mixing 先定义 mixture distribution 如 Dirichlet、regression method 如 linear/gradient-boosted trees、以及基于 downstream eval 的 target——须防过拟合。小/大尺度有 cost–accuracy 差异；两个愿望是 regressor 在 minimizer 附近准确，以及最佳 mixture 从小到大 transfer。

至少有一个明确 scale effect：小模型/少 token 可把 low/high 设0.1/0.9；拿到大模型会把 high epoch 很多而过拟合。Simulated epoching 的通用思想是让 small scale 像 large scale：所有 sources 按 small-run 10B / large-run 1T=0.01 同比例下采样。这样小实验中 epoch 太多的 mixture 也会变差，optimum 更平衡；示例转成 low/high 0.7/0.3。总结：mixing 决定 Wikipedia/general/code 等权重；回归法在小尺度估 mixture→loss 再优化，类似 scaling law；必须处理 epoch/overfit，可用 cap 或 simulation。

**Post-training data 基本 recipe。** 一，定义 environments；二，定义 tasks/prompts；三，从强 teacher 收 responses。

OpenThoughts 用 QwQ-32B teacher 生成1.2M examples，questions 来自27个人类/合成 sources，如 StackExchange、NuminaMath、Chemistry。每 prompt 采16 responses 有帮助；更强 model 不一定是更好 teacher，QwQ-32B 胜 DeepSeek-R1；answer filtering 没帮助；OpenMath-2-Math 等小高质量 source 胜大而 diverse source。讲义展示 sources 与 pipeline。

SWE-smith 给 repo 后让 LM generate tasks，即引入 bugs；128 GitHub repos 得50K tasks。SWE-Zero 指出 SWE tasks dependency 重，不像 math/contest，建立几千 Docker images 是 infrastructure nightmare；强 model 可在无 execution feedback 下解很多 task，说明有 code-semantics 内部 world model。SWE-Zero 有300K无需 repo-specific execution 的 agent trajectories，来自150K GitHub PRs；用 OpenHands scaffold，并删 future git commits 防 agent “git hacking”；由 Qwen3-Coder-480B distill 加 filtering，虽然过滤时仍尽量执行。SWE-Hero 另有13K需要 execution feedback 的 trajectories。

SWE-rebench 有来自3.4K GitHub repos 的21K interactive Python SWE tasks，起点是 GitHub/GitHub Archive 的450K PRs；用 Qwen2.5-72B-Instruct 安装 dependencies、评估 PR quality。SWE-ZERO-12M 把 SWE-Zero 扩至12M agent trajectories，使用 SWE-rebench-v2 的32K executable+120K nonexecutable tasks；运行很小的 mini-coder-1.7B（50.4 pass@100）与 mini-swe-agent，并链接具体 conversation。

Post-training 总结：prompt 可 fully synthetic、真实 environment+合成 task 的 semi-synthetic，或 GitHub PR 等真实；response 来自有能力且适合当 teacher 的模型；code environment 极难维护；仍有大量 filtering 与实现细节。

**全讲总结。** Filtering 通过 language/quality/toxicity classifier 定义“好长什么样”；dedup 用 hash 扩展 fuzzy matching；mixing 在小尺度试 mixture、外推大尺度 optimum；post-training data 长得像 evaluations 并大量使用 synthetic data。许多 data work 依赖具体 domain 与逐例检查。

## 三讲合并成可审计闭环

先用目标评测定义需求，再选择数据源；处理流水线改变实际训练分布；训练后按分域 PPL/能力/安全评测诊断，再回到 mixture/filter。A4 固定模型 recipe，正是为了让这个反馈闭环主要归因于数据。
